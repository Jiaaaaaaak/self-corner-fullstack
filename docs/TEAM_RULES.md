# 專案合作指南 (CONTRIBUTING)

這是 **AI-Classroom** 專案！為了確保專案代碼品質及版本控制的穩定，請所有成員遵守以下開發流程。

---

## 🛠 1. 分支管理規範
本專案採用 **GitHub Flow** 流程。嚴禁直接推送 (Push) 程式碼至 `main` 分支。

- **main**: 穩定版本分支，僅限各組小組長合併 PR。
- **feature/功能名稱**: 個人開發分支（例如：`feature/login-ui`, `feature/stt-api`）。
- **fix/錯誤名稱**: Bug 修復分支。

---

## 🚀 2. 開發與提交步驟

### 第一步：建立分支
在開始開發新功能前，請先切換至最新 `main` 並建立新分支：
`git checkout main` -> `git pull` -> `git checkout -b feature/your-task-name`

### 第二步：開發與 Commit
請定時提交代碼，Commit 訊息請遵循格式：
`feat: 新增 [功能名稱]`, `fix: 修復 [問題描述]`, `docs: 更新文件`

### 第三步：發起 Pull Request (PR)
開發完成後，將分支推送到 GitHub 並發起 PR 指向 `main`。
- **標題格式**：`[前端/後端/基礎設施] 功能描述`
- **內容描述**：請簡述你修改了哪些部分，是否有影響到其他模組。

---

## 🔍 3. 審核與合併機制 (核心規範)

雖然所有成員都擁有 Write 權限，但為了確保品質，**請嚴格遵守「交叉審核」制度**：

### A. 指定審核人 (Reviewers)
發起 PR 後，請務必根據你修改的「資料夾」在右側手動指派審核人：
- **修改 `/frontend/`**：指派 **前端小組長** (@帳號)
- **修改 `/backend/`**：指派 **後端小組長** (@帳號)
- **修改 `/database/`**：指派 **資料庫管理長** (@帳號)
- **修改 `/infrastructure/` 或 `/database/`**：指派 **Git 長** (@帳號)
- **github權限問題**：請tag **Git 長** (@帳號)

### B. 合併權限
1. **組員**：負責撰寫、測試並發起 PR，**禁止自行點擊「Merge pull request」**。
2. **小組長**：收到 PR 通知後進行 Code Review。確認無誤後，請點擊 **Approve**。
3. **最終合併**：由該區域的**小組長**或 **Git 長**執行最後的合併操作，並刪除已合併的分支。

---

## 📜 4. 上傳時程與溝通
- **定時上傳**：建議每日開發結束後將進度推送到個人分支，避免代碼遺失。
- **衝突處理**：若發生 Merge Conflict，請負責該 PR 的成員自行在本地解決衝突後再次推送。
- **重大變更**：涉及資料庫 Schema 或全域配置的變更，請務必先在 LINE 群組或 Issue 中討論。
